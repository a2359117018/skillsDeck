## 核心思路

文字描述 UI 和架构天生有信息损耗 — "简洁现代风格"对你和对 Agent 可能是完全不同的画面。
这个技能通过生成可交互的 HTML 原型，让你直接看到并选择，绕过文字的歧义。

## 触发时机

当对话中出现以下信号时，主动生成视觉选项，不要等用户明确要求：

- 讨论页面布局、组件排列方式
- 比较不同配色或视觉风格
- 设计系统架构、组件层次、数据流
- 用户表达不确定（"不知道怎么弄好"、"你看着办"）
- 任何"文字越描述越糊涂"的时刻

生成选项前先简短告知用户："我来生成几个可视化方案给你选。"

## 两个模式

### UI 模式

适用于：布局选择、配色方案、组件样式、交互效果、动画风格、响应式方案

每个选项应该是一个 **可交互的小原型**：

- 使用真实感的模拟数据（不要用 lorem ipsum）
- 实际的 CSS 样式，不是示意图
- 可点击、可 hover、可展开的交互
- 清晰标注方案名称和核心特点

**配色要求**：每个方案必须使用不同的配色方案，差异要大到一眼可辨。
读取 `references/color-palettes.md` 获取 8 套预设调色板，每次从中选择差异最大的组合。
例如：方案 A 用暖色调色板，方案 B 用冷色调色板，方案 C 用深色主题。

### 架构模式

适用于：系统分层、组件关系、数据流、API 设计、状态管理方案

每个选项应该是一张 **高质量的架构图**：

- 使用 SVG 绘制，不要用 CSS 布局模拟
- 必须用 `<marker>` 定义箭头，不要用文字符号（→、=>）模拟
- 组件/模块用带圆角的矩形（`rx="12" ry="12"`），内部有标题 + 一行描述
- 层级用颜色区分，遵循统一颜色体系（表现层蓝、逻辑层黄、数据层绿、外部紫）
- 标注关键接口和数据格式
- 底部有图例

**架构图质量是重点** — 如果不确定怎么画：

1. 读取 `assets/templates/arch-diagram.html` 获取完整 HTML 模板
2. 读取 `references/arch-diagram-guide.md` 获取颜色编码和绘制规范
3. 将 SVG 内容填充到模板的 `{{SVG_CONTENT}}` 位置

### 模式选择

| 用户在讨论...                    | 模式                           |
| -------------------------------- | ------------------------------ |
| "页面长什么样"、"布局"、"颜色"   | UI 模式                        |
| "怎么组织"、"数据怎么流"、"架构" | 架构模式                       |
| 两者都有                         | 先架构确定结构，再 UI 确定外观 |

## HTML 生成规范

所有选项放在 **一个自包含的 HTML 文件**中（内联 CSS + JS，不依赖外部资源）。

### 页面结构

```
顶部标题栏 → 说明正在展示什么 + 告诉用户如何选择
选项区域 → 2-4 个方案并排展示
底部 → 每个方案的简要说明 + 优缺点对比
```

### UI 模式的 HTML 要点

- 方案之间用明显的分隔线或卡片包裹
- 每个方案有标题（如"方案 A: 暗色极简"）和一句话特点
- 各方案之间要有 **明显的风格差异**，不是微调 — 如果差异太小，用户无法判断偏好
  - 布局差异：不要只是间距微调，要是完全不同的布局结构
  - 配色差异：每个方案用不同色系的调色板（参考 `references/color-palettes.md`）
  - 3 个方案中至少 1 个是深色主题
- 包含真实的交互：按钮 hover、下拉展开、标签切换等
- 使用 CSS 变量（`--primary`, `--bg`, `--surface` 等）方便用户理解配色逻辑
- 视觉品质要高 — 这不是线框图，是接近成品的原型

### 架构模式的 HTML 要点

- 每个架构方案用独立的可视化区域
- **必须用 SVG 绘制**，不要用 HTML div 模拟框图
- 每个方案至少 400px 高度，确保有足够空间展示细节
- 箭头用 `<marker>` 定义，不要用文字箭头
- 颜色编码一致：同类型模块颜色相同
- 每个模块框内要有标题 + 简短描述两行文字
- 底部列出每个方案的核心取舍（如"方案 A：简单但扩展性差，方案 B：复杂但灵活"）
- 如果不确定 SVG 怎么画好看，参考 `references/arch-diagram-guide.md` 获取颜色编码和绘制规范
- **缩放/拖拽/双击交互已包含在 `assets/templates/arch-diagram.html` 中**，无需重复实现

### 通用要求

- 默认浅色背景，除非用户之前表达过深色偏好
- 响应式布局，窗口缩小也能看
- 不依赖任何 CDN 或外部资源（包括字体 — 用系统字体）
- 文件名格式：`[主题]-options.html`
- 代码注释用中文

### 文件管理

**使用 scripts/generator.py 管理所有文件操作**：

```bash
# 生成 HTML 并自动打开浏览器
python visual-options/scripts/generator.py generate --mode arch --title "React状态管理" --content "<svg>...</svg>"

# 仅生成不打开
python visual-options/scripts/generator.py generate --mode ui --title "博客布局" --content "<html>...</html>" --no-open

# 清理临时文件（用户选完后执行）
python visual-options/scripts/generator.py cleanup
```

**文件名格式**：`[时间戳]-[主题]-[模式]-options.html`

- 自动生成，无需手动指定
- 示例：`1778406275-测试架构图-arch-options.html`

**流程**：

1. 生成后自动打开浏览器，用户在浏览器中选择方案
2. 用户选择后，Agent 将选定方案内容提取到项目合适位置
3. 执行 `cleanup` 删除临时文件，告知用户最终保存路径

## 选择与迭代流程

```
用户描述需求
    ↓
生成 2-4 个视觉选项
    ↓
用户选择一个方案
    ↓
判断：是否需要细化？
    ↓           ↓
  需要细化    可以开始实现
  （基于选择   ↓
   生成更      保存偏好
   精细的      → 开始编码
   子选项）
```

### 收到选择后

1. **确认选择**：复述用户选了哪个方案及其关键特征
2. **判断细化需求**：
   - 方案已足够具体 → 直接提议开始实现
   - 还有未确定的细节 → 问用户："这个方案需要进一步细化吗，还是直接开始实现？"
3. **不要过度询问** — 如果用户很果断，直接行动
4. **清理文件**：
   - 将选定方案内容提取到项目合适位置（如 `src/components/`、`docs/`）
   - 删除 `.visual-options/` 目录
   - 告知用户最终文件保存路径

### 细化迭代

如果需要细化，基于已选方案生成 2-3 个更精细的子选项：

- **保留**用户已选定的所有特征
- **只在未确定的维度**上提供变化
- 比上一轮更精细、更接近最终产品
- 最多细化 2 轮，避免无限循环

## 偏好记忆

当用户做出明确的设计选择后，将偏好保存到项目记忆中。

保存内容：

- 选择的方案类型和特征
- 用户明确表达过的偏好（如"我喜欢简洁的"、"不要太花哨"）
- 偏好的配色风格（如"偏冷色"、"喜欢深色主题"）
- 不喜欢的风格（如"太传统"、"老式风格"）
- 适用的场景（如"dashboard 类页面"）

保存位置：使用 memory 系统的 `reference` 类型。

后续同类设计时：

- 将已保存的偏好作为默认方案的起点
- 同时仍提供 1-2 个不同风格的选项供对比
- 告知用户："根据你之前的偏好，我默认采用了 X 风格，同时也有其他方案可选"

## 参考文件

当生成选项时，按需读取以下参考文件：

- `references/arch-diagram-guide.md` — 架构图绘制规范（颜色编码、marker 定义、好的/差的图特征）
- `assets/templates/arch-diagram.html` — 架构图 HTML 模板（包含缩放/拖拽/双击交互 JS）
- `references/color-palettes.md` — 8 套预设配色方案

不需要每次都读 — 但如果对配色不满意或架构图画不好，务必先读参考再生成。

## 示例场景

### 场景 1：用户说"帮我设计一个用户管理页面的布局"

→ 检测到 UI 模式
→ 从 color-palettes.md 选 3 套差异大的调色板（如极光蓝、暗夜深蓝、奶油极简）
→ 生成 3 个选项：卡片网格（暖色）、表格式列表（冷色）、侧边栏详情（深色）
→ 每个都是可交互的小原型，有模拟用户数据

### 场景 2：用户说"这个前端项目的状态管理怎么设计"

→ 检测到架构模式
→ 读取 `assets/templates/arch-diagram.html` 获取 HTML 模板
→ 读取 `references/arch-diagram-guide.md` 获取 SVG 绘制规范
→ 生成 2-3 个架构图：Redux 集中式、Zustand 轻量级、Context 分散式
→ 每张图用高质量 SVG 展示组件关系和数据流，有图例和标注

### 场景 3：用户说"dashboard 既要好看又要数据清晰"

→ 混合模式：先架构（布局分区方案）→ 选定后 → UI（配色和组件样式）
→ 分步生成，每步都让用户选择
